这个 scene 它其实是单个的 ability 。我们要做的是构建一个包含多个的有dummymain的这样的 create 的函数。就是试着要把所有的scene都结合起来，并且这些 scene 之间，它的不同 entry point 里边其实是有个跳转的关系的。
我们这个时候更加希望能够，比如说我 ability a 能够跳转到 b。ability B 能够跳转到 ability C。这样的一个跳转关系，我们希望能够在这样的一个 cfg 里边有进行体现。
其实要把 flowdroid里边的这样的一个安卓的生命周期给它模拟进来。其实我们可能要扩展的就是这个 dummy main creator 
然后这个 main 的话，我们要在它这个基础上面去把那个鸿蒙的生命周期给它模拟进去。这样的话，它构建的 cfg 其实就是包含了整个我们构造的这样的一个 main 函数的，包含了多个 ability 之间的启动，关闭以及之间相互跳转的这样的一个完整的控制流。这个其实就是我们要做的事情。
说到底就是把这样的一个 dummy main creator 。它也构建了一个就是虚拟的 main 函数。但是这个虚拟的 main 函数其实只包含了单个的 scene，就是我们的 UI ability 等等之类的，就是单个的 鸿蒙 class。
那么这个和我们之前看过的 flowdroid的代码一样。实际情况下面它其实不是一个单个的 UI ability，而是多个 UI ability，并且这些多个 UI ability 之间可以相互跳转的一个状态。所以其实我们要做的就是这个事情。
比如说：我去找到一个整个应用的，就是主要开始应用开启的时候。我这样的一个 class 是什么样的 class 。然后这个 class 它对应的这个函数跳转是什么样子的，就是它可以导向哪些其他的 uiability，然后这些 uiability 它上面有哪些这样的一个生命周期函数，它可以逐步的去调用。调用的过程中，它页面展示了之后，显示了哪些控件。这些控件上面的这个回调函数是如何去调用的，然后把这样的一个流程，把它全部建模在这个dummy main的这个函数里边，这个就是我们要去做的一个事情。
剩下的我们只要复用它的这样的一个整个的控制流的构建，复用它的这个数据流的构建，然后用它这样的一个 ifds 算法去求解就可以了。然后我们其实就能得到一个基础的污点分析框架。
就是下面一步，我们首先可以看到的是我们的这个 arkanalyzer 以及我们之前学习过的这个 flowdroid的，它代码是非常相近的。包括它的一个 cfg 构建以及它的数据流，它也提供了 ifds 算法。但是不同的一点：它也虽然也构建了这样一个dummy main creator ，就是构建了一整个应用的静态分析的一个入口函数。但是它的入口函数是以scene为单位的，它是以某一个 uiability 作为它的主要分析内容，它并没有考虑我页面是如何跳转的，我页面之间它是怎么样去进行。
回调函数其实是做的是比较粗糙的，就可以看到它整个 ability ,有哪些 oncreate onclick ontouch 以及 ondrag 等等之类的。这个回调函数要直接去调用它回调函数，那我们可能要我们就要做的更加精细一点，要做的就比如我当前的这个 uiability 所继承的这些 onclick 方法，它这个 uiability 上面所有遇到的控件，它上面的这个 onclick 方法，我都要根据它这个控件去进行实例化。实例化了之后再去进行调用，去进行就在这个 creator 里边去提供一个函数的调用接口。
就这部分，这之前可能先同学们可能先自己去试着去写一写，就包括怎么去拓展dummy main creator 可以仿照 flowdroid的方式去进行扩展。
最简单的方式，就是我先把这个整个项目里边有多少这个 scene。  我先把它全部放到我的 dummy main creator 里边。然后我让它去逐步生成这个 scene 里边的所有的，比如说 oncreate onstart onstop 它的生命周期函数全部执行一遍，包括了我这个 scene 里边所包含的它这个，比如说控件。比如说它的 button，它的 text view 以及这些控件上面的这个 on click 方法 on stop on click 方法 on touch 方法我都建模在这样的一个 dummymain 里边，其实就是去扩充它这个 dummymain creator 我们可能要继承，或者说是要扩展一下 dummy main creator。
